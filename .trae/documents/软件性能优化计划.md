# 软件性能诊断与优化计划

经过深入的代码审查，我们定位到了导致软件卡顿的三个核心原因，并制定了相应的优化方案。

## 核心问题诊断

1.  **AI 任务阻塞主进程 (严重)**
    *   **现象**：应用界面在进行搜索或 AI 分析时会完全冻结，无法响应点击。
    *   **原因**：`ragEngine.js` 在 Electron 主进程中同步运行 Transformer 模型（Embedding 生成和 Reranking），这会耗尽 CPU 并阻塞 UI 渲染。
    *   **位置**：`electron/services/rag/ragEngine.js`

2.  **多维表格渲染瓶颈 (中等)**
    *   **现象**：表格数据较多时，滚动和编辑出现延迟。
    *   **原因**：`MultidimensionalTable.tsx` 直接渲染所有数据行（HTML `<table>`），随着数据量增加，DOM 节点数量呈线性增长，造成浏览器重排/重绘压力。
    *   **位置**：`components/MultidimensionalTable.tsx`

3.  **流程图组件无效重绘 (中等)**
    *   **现象**：拖拽流程图节点时不够流畅，帧率下降。
    *   **原因**：`ExecutionNode` 组件未使用 `React.memo` 包裹，导致每次父组件更新时，所有节点都会被迫重新渲染。
    *   **位置**：`components/WorkflowEditor.tsx`

## 实施方案

我们将分三步解决上述问题：

### 第一步：后端架构解耦 (RAG Engine)
将计算密集型任务移出主进程，利用现有的 Worker 线程。

*   **修改 `electron/services/rag/ragWorker.js`**:
    *   增加 `query` 消息处理器：接收文本，返回向量。
    *   增加 `rerank` 消息处理器：接收查询和候选文档，返回排序结果。
*   **修改 `electron/services/rag/ragEngine.js`**:
    *   移除直接的模型调用。
    *   实现与 Worker 的异步通信机制（Request-Response 模式），将任务分发给 Worker 执行。

### 第二步：流程图性能优化
减少不必要的 React 组件渲染。

*   **修改 `components/WorkflowEditor.tsx`**:
    *   使用 `React.memo` 包裹 `ExecutionNode` 组件，确保只有数据变化时才重绘。

### 第三步：表格虚拟化
引入虚拟滚动技术，大幅降低 DOM 节点数量。

*   **修改 `components/MultidimensionalTable.tsx`**:
    *   引入 `react-window` 和 `react-virtualized-auto-sizer`。
    *   将 `<table>` 结构改造为适合虚拟化的 `<div>` 结构（或保持 Table 结构并使用 `innerElementType`）。
    *   实现只渲染可视区域内的行。

## 预期效果
*   **零卡顿搜索**：AI 任务在后台线程运行，主界面保持丝般顺滑。
*   **流畅滚动**：无论表格有多少行数据，滚动都能保持 60fps。
*   **高效交互**：流程图拖拽和编辑不再有延迟感。
