# 全域助手功能诊断与优化清单

通过对 `GlobalAIAssistant.tsx` 及底层服务 `geminiService.ts` 的深入代码分析，我为您整理了当前的功能现状以及针对性的优化建议。

## 1. 当前功能清单 (Status Quo)

目前的全域助手（悬浮球）本质上是一个**具备基本路由能力的对话/检索窗口**。

### ✅ 已实现功能
*   **基础交互**：
    *   **悬浮球形态**：支持拖拽移动、悬浮动画、点击展开/收起。
    *   **全息窗口**：毛玻璃 UI、对话气泡、Markdown 渲染、设置面板（仅清理缓存）。
*   **智能中枢 (Orchestrator)**：
    *   **意图识别**：通过 AI 分析用户输入，自动分类为 "System"（查系统）、"Assistant"（查知识库）或 "General"（闲聊）。
    *   **上下文感知**：(刚刚修复) 支持读取最近 3 条对话历史，理解指代性问题。
*   **数据能力**：
    *   **系统快照**：能回答关于“当前有哪些项目”、“下周有什么日程”的问题（原理是将**全量**项目和日程数据转为文本喂给 AI）。
    *   **RAG 检索**：能调用特定的知识库助手（如财务助手），检索本地向量数据库并回答。

### ❌ 未启用/缺失的潜在能力
虽然底层 `geminiService.ts` 已经具备了强大的功能，但全域助手 UI 层**尚未接入**，导致用户无法通过助手球使用这些能力：
*   **无法控制导航**：虽然接收了 `onNavigateToModule` 接口，但从未调用。用户说“打开日历”时助手无法跳转。
*   **不支持多模态**：没有文件上传或拖拽入口。用户无法拖入发票进行识别，也无法发送图片。
*   **无高级工具调用**：无法使用底层已有的“一句话立项”、“海报生成”、“任务拆解”等高级 AI 工具。
*   **无主动操作**：只能“查”，不能“改”。例如无法通过对话创建日程或任务。

## 2. 优化方案 (Roadmap)

基于上述分析，建议按照以下优先级进行强化和增删：

### 🔥 需强化的功能 (High Priority)
1.  **实现语音控制/导航 (Navigation Control)**
    *   **现状**：接口已预留但闲置。
    *   **方案**：在路由逻辑中增加 `navigation` 意图。当用户说“去日历”或“打开财务看板”时，直接调用 `onNavigateToModule` 跳转页面。
2.  **增强系统查询 (Function Calling)**
    *   **现状**：一次性通过 Context 注入所有数据。如果项目多了，Token 会超限且更新不及时。
    *   **方案**：改为 Tool Use 模式。AI 决定需要查项目时，再动态调用 `getProjects()`，实现按需获取。

### ➕ 建议增加的功能 (Additions)
1.  **多模态文件投喂 (File Drop Zone)**
    *   **功能**：允许用户直接拖拽文件（PDF/图片）到悬浮球上。
    *   **场景**：拖入发票直接触发 `analyzeReceipt`（底层已实现）；拖入文档直接进行摘要。
2.  **快捷指令面板 (Quick Actions)**
    *   **功能**：在对话框底部添加常用指令按钮，如“生成周报”、“分析当前项目”、“创建会议”。
    *   **价值**：降低用户通过自然语言探索功能的门槛。

### ✂️ 建议剔除/修改的功能 (Deprecations/Changes)
1.  **全量数据注入 (Context Injection)**
    *   **剔除**：停止在每次对话中默认注入所有 `projects` 和 `events` 数据。
    *   **替代**：改为上述的按需 Tool Calling 或只注入摘要信息，提升响应速度和隐私性。
