# RAG 架构升级：自适应分级检索方案

您提出的“分级处理”思路非常精准。当前的 RAG 引擎确实存在“杀鸡用牛刀”的问题——无论问题多简单（如“有多少个文件”），都会走一遍完整的向量检索+重排+大模型生成，既慢又笨重。

我将按照您的建议，实施**自适应分级检索架构**，引入“快慢路由”机制。

## 🏗️ 新架构设计

### 1. 智能路由器 (The Router)
在任务开始前，先对用户意图进行毫秒级判断，将请求分流到三条路径：

*   **⚡️ 快速通道 (Fast Path)**：针对检索、计数、清单类问题。
    *   *特征*：“列出所有PDF”、“有多少关于预算的文件”、“查找2024年的合同”。
    *   *执行*：**不调用向量模型，不生成长文**。直接使用 SQL/代码在元数据中检索。
    *   *反馈*：直接返回结构化数据（表格、列表、数字）。
*   **🧠 深度通道 (Slow Path)**：针对分析、总结、创作类问题。
    *   *特征*：“分析这个项目的风险”、“总结会议纪要”、“根据材料写一份大纲”。
    *   *执行*：走完整的 RAG 流程（向量检索 -> 重排 -> 上下文组装 -> LLM 推理）。
    *   *反馈*：流式输出长文本，配合引用溯源。
*   **🔄 歧义处理 (Ambiguity Handler)**：针对模糊问题。
    *   *特征*：“那个文件怎么说？”（没指明哪个文件）。
    *   *执行*：LLM 重写问题 -> 自动补充上下文 -> 重新路由。

### 2. 实施步骤

#### 第一阶段：增强意图识别 (Router)
*   升级 `ragEngine.js` 中的 `analyzeQueryIntent`，增加 `structured_search`（结构化检索）分类。
*   让它能提取关键过滤条件（如：`file_type: pdf`, `date_range: 2024`, `keyword: "预算"`）。

#### 第二阶段：构建快速执行器 (Fast Executor)
*   在 `vectorStore.js` 中封装“元数据查询”接口（基于 SQL 的 `LIKE` 或 `metadata` 过滤）。
*   在 `ragEngine.js` 中新增分支逻辑：一旦识别为快速任务，直接调用数据库查询接口，并用极简模版输出结果（例如：“共找到 5 个相关文件：...”）。

#### 第三阶段：前端反馈优化
*   针对“快速通道”的结果，在聊天界面展示更紧凑的卡片（而不是大段文字）。
*   针对“深度通道”，保持现有的进度条（Execution HUD）反馈，缓解等待焦虑。

## 🚀 预期效果
*   **提速**：简单找文件/找数据的任务响应时间从 ~10秒 缩短至 ~1秒。
*   **准确**：避免 LLM 在简单计数问题上出现幻觉（LLM 不擅长数数，代码擅长）。
*   **体验**：用户会感觉系统变“聪明”了，知道什么时候该快，什么时候该深思熟虑。
