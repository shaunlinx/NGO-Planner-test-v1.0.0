# 修复与升级计划：注入系统级 Agent 与增强 AI 逻辑

您看到的 `Agent not found` 错误是因为我们预设的模版使用了特定的 Agent ID（如 `agent-copywriter`），但这些 ID 并没有在您的团队成员列表 (`teamMembers`) 中定义。为了解决这个问题并确保“跑通功能”，我们将进行以下关键修复：

## 1. 注入系统级 Agent (System Agents Injection)
在 `MultidimensionalTable.tsx` 中，我们将修改 `experts` 的生成逻辑。不再仅仅依赖用户手动添加的团队成员，而是**动态注入**模版所需的系统级 Agent。这样无论用户团队配置如何，模版都能立即找到对应的“执行者”。

## 2. 增强 Agent 执行逻辑 (Enhanced Execution)
当前的 `runAgentTask` 逻辑中，如果检测到 API 可用，会优先尝试调用真实 LLM。但为了保证在**演示环境**下的稳定性和视觉效果（如海报生成），我们将优化逻辑：
- **混合模式**：对于“视觉助手”和“发布助手”，我们将优先使用**高级模拟逻辑**（返回图片和带格式的日志），因为纯文本 LLM 无法生成图片或操作微信后台。
- **兜底逻辑**：确保在任何网络环境下，Agent 都能返回有意义的结果，而不是卡死或报错。

## 3. 实施步骤
1.  **修改 `experts` 计算逻辑**：在组件内部硬编码注入 `agent-copywriter`, `agent-designer` 等 7 个系统角色。
2.  **优化 `runAgentTask`**：调整 Mock 逻辑的优先级，确保特定 ID 的 Agent（如 Designer）总是返回图片，而不是尝试去调用 LLM 生成一段描述文字。

完成这两步后，您点击“运行”时，表格将不再报错，而是立即生成文案、海报和发布日志，完美展示工作流闭环。
